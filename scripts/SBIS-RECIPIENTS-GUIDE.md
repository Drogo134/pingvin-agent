# Кого назначить получателем обращения в Saby

Кратко: **получатель — не «любой сотрудник из справочника»**, а тот, кого **разрешил регламент «Обращение»** на этапе **«Отправка»** (кнопка «На выполнение» / API `Выполнить`).

Проверено API (обращение №85): отклонены **все** варианты — ИИ Агент `27850`, Ткачук `223`, Федоров `3374`, Васьков `30`, подразделение «РПК ПИНГВИН, ООО». Значит, в регламенте **не настроена очередь получателей** (или `ai_agent` не может назначать этих людей).

---

## Три рабочих схемы (что выбрать)

### Вариант A — **общая очередь** (рекомендуется для ПинГвин)

**Смысл:** заявки от бота видят **все менеджеры** подразделения, кто в очереди Контакт-центра.

| В `.env` | Значение |
|----------|----------|
| `SBIS_RECIPIENT_MODE` | `auto` или `department` |
| `SBIS_DEPARTMENT_NAME` | `РПК ПИНГВИН, ООО` |
| `SBIS_DEPARTMENT_ID` | **не задавать** (id `4` из кадров — не очередь ДО) |
| `SBIS_AUTO_SUBMIT` | `true` |

**В Saby (админ):** в регламенте «Обращение» на этапе «Отправка» в список получателей добавить **подразделение «РПК ПИНГВИН, ООО»** (или роль «операторы контакт-центра»).

---

### Вариант B — **один живой менеджер**

**Смысл:** все обращения от бота падают **одному** сотруднику (например, старшему менеджеру).

| В `.env` | Значение |
|----------|----------|
| `SBIS_RECIPIENT_MODE` | `manager` |
| `SBIS_MANAGER_ID` | id **реального** оператора с доступом в Saby |

**Кого можно указать (из `python scripts/sbis_discover.py`):**

| ID | ФИО | Доступ в Saby | Как получатель |
|----|-----|---------------|----------------|
| `223` | Ткачук Андрей | Да | Кандидат — **после** настройки регламента |
| `3374` | Федоров Иван | Да | Кандидат — **после** настройки регламента |
| `27850` | ИИ Агент | Да | **Только логин API** (`SBIS_LOGIN`), **не** очередь заявок |
| `30` и др. | Васьков и др. | Нет | Ошибка «работающий сотрудник» |

```env
SBIS_RECIPIENT_MODE=manager
SBIS_MANAGER_ID=223
SBIS_AUTO_SUBMIT=true
```

**В Saby:** этот сотрудник должен быть в **очереди получателей** регламента и иметь права [Контакт-центр / Обращение](https://saby.ru/help/another/helpdesk/law_appeal).

---

### Вариант C — **контракт** (маршрутизация по клиентам)

**Смысл:** в [Контакт-центр → Обращения → Клиенты](https://online.sbis.ru/page/support-service) создаётся **контракт**, в него — клиенты, на контракте закрепляется **ответственный оператор**. Новые обращения этого клиента идут к нему.

Подходит, если разные менеджеры ведут разных клиентов. Для Telegram-бота с «анонимными» клиентами чаще проще **вариант A**.

Документация: [Создать контракт](https://saby.ru/help/another/helpdesk/sk), [Добавить клиента в контракт](https://saby.ru/help/another/helpdesk/klient).

API бота контракт **не подставляет автоматически** — нужна привязка клиента в Saby.

## Кто есть кто (чтобы не путать)

| Роль | Кто | В `.env` |
|------|-----|----------|
| **Автор API** | ИИ Агент / `ai_agent` | `SBIS_LOGIN`, `SBIS_PASSWORD` — создаёт черновики |
| **Получатель** | Живой менеджер или подразделение из очереди регламента | `SBIS_MANAGER_ID` или `SBIS_DEPARTMENT_NAME` |
| **Уведомления Telegram** | Chat ID менеджеров | `MANAGER_TELEGRAM_CHAT_IDS` — параллельно, всегда работает |

**ИИ Агент не должен быть получателем** — он техническая учётка. Заявки должны идти людям (Ткачук, Федоров) или в общую очередь «РПК ПИНГВИН, ООО».

---

## Как узнать «правильного» получателя из Saby (без угадывания)

1. Админ в UI: **Контакт-центр → + → Обращение** (или открыть черновик бота).
2. Нажать **«На выполнение»** → в списке появятся **допустимые** сотрудники/подразделения — это и есть белый список регламента.
3. Выбрать того, кому должны падать заявки от бота.
4. Скопировать `guid` из URL обращения после отправки.
5. На ПК:

```powershell
python scripts\sbis_show_recipient.py <guid>
```

6. Перенести в `.env` то, что покажет скрипт (`Подразделение` или `Ответственный` / `Исполнитель` на этапе).

## Обходные пути (пока регламент не настроен)

| Путь | Что даёт | Ограничение |
|------|----------|-------------|
| **Telegram** `notify_managers.py` | Менеджеры видят ТЗ и ссылку `open_url` | Не статус в очереди Saby |
| **Черновик + ссылка** | `create_appeal` работает, № в Saby | В списке «Ожидается отправка» |
| **Ручное «На выполнение»** | Менеджер в UI выбирает получателя | Не автоматизация |
| **API `Решено`** | Закрыть без получателя | Сейчас блокируется **обязательными полями** формы (вид обращения, клиент, описание) |

## Что попросить администратора Saby (чеклист)

1. **Права `ai_agent`:** [Права на обращения](https://saby.ru/help/another/helpdesk/law_appeal) — участки «Служба поддержки» и «Обращение», **Просмотр и изменение**; «Обработка документов за других» — **Разрешено**.
2. **Регламент «Обращение»** → этап **«Отправка»** → кто может быть **получателем** при действии «Выполнить»: подразделение **РПК ПИНГВИН, ООО** и/или сотрудники **223**, **3374** (или ваша очередь).
3. **Опционально:** ослабить обязательность полей при закрытии из API или завести [доп. поля](https://saby.ru/help/integration/api/sequence/add_fields) с именами из редактора регламента.
4. После настройки:

```powershell
python scripts\sbis_try_all_recipients.py
python scripts\sbis_submit_drafts.py
.\scripts\start-local.ps1
```
